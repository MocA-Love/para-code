# Claude hook version probe recovery design

## 背景

Para Code は shared process の起動時に `claude --version` を実行し、確認できた Claude Code のバージョンに応じて追加 hook を `~/.claude/settings.json` へ登録する。現在は2秒以内にバージョン取得できなかった場合、その失敗結果をプロセス存続中キャッシュする。このため、一時的な起動遅延や PATH 解決の問題があると、60秒ごとの hook 監査でも追加 hook が登録されない。

## 目的

- Claude Code の実行ファイルとバージョンを GUI 起動環境でも確実に取得する。
- 一時的な取得失敗から自動回復する。
- 永続的な失敗では再試行を有限回で打ち切る。
- バージョン不明時に未確認の hook を登録しない既存の安全側の挙動を維持する。
- 取得失敗の原因をログから判別可能にする。

## 対象範囲

`ParadisAgentHooksReconciler` が行う Claude Code のバージョン取得と、その結果のキャッシュ・再試行・ログ出力を変更する。hook のイベント一覧、最低対応バージョン、JSONマージ、ファイル監視、Codex hook 登録は変更しない。

## 設計

### 実行ファイルの解決

1. 既存の `ParadisCachedShellEnv` からログインシェル由来の環境を取得する。
2. VS Code 共通の `findExecutable()` に解決済み環境を渡し、`claude` の絶対パスを取得する。
3. 実行ファイルが見つからない場合は、その試行を失敗として記録する。継承PATHだけを使った暗黙の再実行や、固定パスへのフォールバックは行わない。

### バージョン取得

解決した絶対パスに `--version` を渡して直接実行する。タイムアウトは15秒とする。終了コード0で、既存のバージョン解析が受け付ける出力を得た場合だけ成功とする。

### 再試行とキャッシュ

- 最大試行回数は初回を含め3回とする。
- 初回は reconciler 起動時に実行する。
- 失敗後の再試行は、既存の60秒定期監査時に1回ずつ行う。
- 失敗結果は成功キャッシュへ保存しない。
- 3回失敗した後は、その shared process の存続中は追加試行しない。
- 取得に成功した場合はバージョン出力をプロセス存続中キャッシュし、それ以降は再実行しない。
- 再試行中も基本 hook の整合処理は継続する。
- 成功した監査回で、対応する追加 hook を既存の冪等マージ処理によって登録する。

### 安全側の挙動

バージョン取得に成功するまでは、現在と同じ基本 hook だけを管理対象とする。バージョン不明を理由に、`MessageDisplay` や activity hook を推測登録しない。

### ログ

各失敗時に、試行番号と失敗段階を warning として記録する。

- shell environment resolution failure
- executable not found
- process spawn error
- timeout
- non-zero exit code
- unparseable version output

ログには認証情報や環境変数全体を含めない。実行ファイルパス、終了コード、signal、エラーコード、制限した stderr のみを診断情報として扱う。3回目の失敗では、そのプロセス中の再試行を打ち切ったことも記録する。成功時は、取得バージョンと有効になった管理イベントを既存の info ログへ記録する。

## テスト

次の単体テストを追加する。

1. 解決済みPATHから Claude 実行ファイルを取得し、バージョン取得に成功する。
2. 1回目の失敗後、次の監査で再試行して成功し、追加 hook が登録される。
3. 3回連続失敗した後は、以降の監査でプローブを実行しない。
4. 失敗中も基本 hook の整合が継続する。
5. 成功後はバージョンプローブを再実行しない。
6. 実行ファイル未検出、タイムアウト、非ゼロ終了、解析不能出力を区別して扱う。
7. バージョン境界と追加 hook の既存テストが引き続き成立する。

外部の Claude Code バイナリには依存せず、実行ファイル解決とプロセス実行を注入可能にしてテストする。

## 完了条件

- 起動時の一時的な失敗後、最大3回の範囲で自動回復する。
- Claude Code 2.1.207の取得成功後、対象の追加 hook が登録される。
- 取得失敗時は追加 hook が登録されない。
- 3回失敗後は不要なプロセス起動を繰り返さない。
- ログから失敗段階と試行回数を判別できる。
- 既存ユーザーhookを保持する冪等マージ動作に変化がない。
